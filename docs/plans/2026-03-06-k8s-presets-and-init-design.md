# K8s Deployment Simplification: Helm Presets + `ax k8s init`

**Date:** 2026-03-06
**Status:** Draft

## Problem

The AX Helm chart's `values.yaml` is 230+ lines and growing. Deploying AX to
Kubernetes requires understanding NATS, PostgreSQL, sandbox tiers, pool
controllers, network policies, and more. The target audience ŌĆö application
developers who barely know K8s ŌĆö should be able to deploy with minimal
configuration.

## Decision

Instead of building a Kubernetes operator (high complexity, large engineering
investment), we simplify the existing Helm chart with two complementary features:

1. **Helm presets** ŌĆö a `preset` key that collapses all configuration into
   sensible defaults for three deployment sizes.
2. **`ax k8s init` CLI command** ŌĆö an interactive wizard that generates a values
   file and creates K8s secrets.

No operator. No CRDs. No new runtime components.

## Design

### Helm Presets

A new top-level `preset` value in `values.yaml`. When set, it provides defaults
for replica counts, resource limits, sandbox tiers, NATS sizing, and PostgreSQL
configuration. User overrides take precedence over preset values.

**Resolution order:** user override > preset default > chart default

#### Preset Definitions

| | `small` | `medium` | `large` |
|---|---|---|---|
| Host replicas | 1 | 2 | 3 |
| Agent-runtime replicas | 1 | 3 | 5 + autoscale |
| Sandbox tiers | light only (1-3 pods) | light (2-10) | light (2-10) + heavy (0-3) |
| PostgreSQL | internal subchart | external | external |
| NATS | 1 replica, no cluster | 3 replicas, clustered | 3 replicas, clustered |
| Network policies | enabled | enabled | enabled |
| Pool controller | enabled | enabled | enabled |
| Sandbox runtimeClass | "" (no gvisor) | gvisor | gvisor |
| Host resources | 250m/256Mi | 500m/512Mi | 500m/512Mi |
| Agent-runtime resources | 500m/1Gi | 1cpu/2Gi | 1cpu/2Gi |

When `preset` is empty (the default), behavior is identical to today ŌĆö full
manual configuration. The existing values.yaml remains as documentation of all
available knobs.

#### Generated Values File

```yaml
# Generated by: ax k8s init
preset: small
global:
  imageRegistry: registry.mycompany.com/ax
  imagePullSecrets:
    - ax-registry-credentials
postgresql:
  external:
    enabled: true
    existingSecret: ax-db-credentials
    secretKey: url
  internal:
    enabled: false
```

#### What the CLI Creates via kubectl

- `ax` namespace (if it doesn't exist)
- `ax-registry-credentials` ŌĆö docker-registry type secret
- `ax-api-credentials` ŌĆö with the appropriate key (e.g., `anthropic-api-key`)
- `ax-db-credentials` ŌĆö with `url` key (only for external DB)

#### What the CLI Does NOT Do

- Run `helm install` (user does that)
- Run DB migrations (host runs them automatically on startup)
- Configure ingress/domain (port-forward only for now)

#### Non-Interactive Mode

All questions can be passed as CLI flags for scripting and CI/automation:

```bash
ax k8s init \
  --preset small \
  --registry-url registry.mycompany.com/ax \
  --registry-user deploy-bot \
  --registry-password "********" \
  --llm-provider anthropic \
  --api-key "sk-ant-********" \
  --database internal \
  --namespace my-namespace \
  --output my-values.yaml
```

| Flag | Interactive equivalent | Required |
|------|----------------------|----------|
| `--preset` | Cluster size prompt | Yes |
| `--registry-url` | Docker registry URL | No (skip imagePullSecret if omitted) |
| `--registry-user` | Registry username | Only with `--registry-url` |
| `--registry-password` | Registry password | Only with `--registry-url` |
| `--llm-provider` | LLM provider prompt | Yes |
| `--api-key` | API key prompt | Yes |
| `--database` | Database prompt (`internal` or `external`) | Yes |
| `--database-url` | PostgreSQL connection URL | Only with `--database external` |
| `--namespace` | Target namespace | No (defaults to `ax`) |
| `--output` | Output file path | No (defaults to `ax-values.yaml`) |

When all required flags are provided, the CLI runs without prompting. If any
required flag is missing, it falls back to interactive prompting for that
question only.

### Database Initialization

No separate DB init job or Helm hook is needed. AX runs migrations
automatically on host startup. Each subsystem (storage, audit, memory,
scheduler, files, orchestration) uses `runMigrations()` from
`src/utils/migrator.ts` to apply pending Kysely migrations when its provider
initializes. The migrations use database-level locking, so concurrent host
replicas are safe.

The only prerequisite is that the PostgreSQL database exists and is reachable:
- `small` preset (internal subchart): handled automatically by the Bitnami
  PostgreSQL subchart
- `medium`/`large` preset (external): user provides the connection URL

- `charts/ax/templates/_presets.tpl` ŌĆö preset definitions and `ax.preset.value`
  resolution helper
- `src/cli/k8s-init.ts` ŌĆö the `ax k8s init` interactive command

### Modified Files

- `charts/ax/values.yaml` ŌĆö add `preset: ""` at top, add
  `global.imagePullSecrets` support
- `charts/ax/templates/_helpers.tpl` ŌĆö add `ax.preset.value` helper
- `charts/ax/templates/host/deployment.yaml` ŌĆö use preset-aware helpers for
  replicas, resources
- `charts/ax/templates/agent-runtime/deployment.yaml` ŌĆö same
- `charts/ax/templates/pool-controller/deployment.yaml` ŌĆö same
- `charts/ax/templates/pool-controller/configmap-sandbox-templates.yaml` ŌĆö
  sandbox tier from preset
- `src/cli/index.ts` ŌĆö register the `k8s init` subcommand

### Not Changed

- No new npm dependencies (CLI uses Node's `readline`, shells to `kubectl`)
- No changes to the migration system
- No changes to host/agent runtime code
- Network policies, RBAC, service, ingress templates stay as-is

## Edge Cases

**Preset + manual override conflicts:** Resolution order (user > preset >
chart default) handles this cleanly. Explicit values always win.

**`kubectl` not installed:** CLI checks at startup, exits with a clear message.

**Namespace already exists:** `kubectl create namespace` failure is caught and
the CLI continues.

**Secret already exists:** CLI warns and asks whether to overwrite or skip.

**Internal PostgreSQL on `small`:** Bitnami subchart handles its own
initialization. AX migrations run on first host startup. No manual steps.

## Alternatives Considered

### Kubernetes Operator

A full operator with CRDs (`AxCluster`, etc.) would provide maximum
abstraction but requires massive engineering investment: CRD versioning, upgrade
logic, operator lifecycle management, and Go expertise (most K8s operators are
written in Go). Overkill for the stated goal of simplifying deployment for
developers who barely know K8s. An operator can be revisited later if
multi-tenancy or complex day-2 operations become a real need.

### CLI-Only (No Presets)

An `ax k8s init` command without Helm presets would still generate a full
values file. This works but doesn't help users who want to hand-edit values or
use GitOps workflows. Presets provide value independent of the CLI.
